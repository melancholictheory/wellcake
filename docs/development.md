# Development

## Prerequisites

- Go 1.24+
- Kubebuilder 4.x
- `make`, `kubectl`, `helm` v3+
- A Kubernetes cluster (kind / k3d / real)

## Layout

```
api/v1beta1/                  — CRD types for the hub/storage version (ValkeyCluster, ValkeyACL, ...) — edit here
api/v1alpha1/                 — served spoke + conversion to the hub (mirror new fields here)
cmd/main.go                   — manager bootstrap
internal/controller/          — controllers and builders
  valkeycluster_controller.go    main reconciler + dispatcher + finalizer
  resources.go                   shared builders (Service, CM, STS, PVC, PDB, NP, Secret)
  cluster.go                     Cluster topology + bootstrap/scale Jobs
  sentinel.go                    Sentinel topology
  failover.go                    operator-driven failover (go-redis)
  backup.go                      S3 CronJob builder
  valkeyacl_controller.go        ValkeyACL reconciler
internal/webhook/v1beta1/     — validating + mutating admission webhooks
config/                       — kustomize manifests for `make install` / `make deploy`
config/crd/bases/             — generated CRDs (do not edit)
config/rbac/                  — generated RBAC (do not edit)
charts/                       — Helm charts
  valkey-operator/              operator Deployment + RBAC + CRDs under crds/
  valkey-cluster/               consumer wrapper for a single ValkeyCluster CR
docs/                         — what you are reading
```

Files generated by kubebuilder must not be edited by hand:

- `api/v1beta1/zz_generated.deepcopy.go` and `api/v1alpha1/zz_generated.deepcopy.go`
- everything under `config/crd/bases/`
- `config/rbac/role.yaml`

Regenerate after changing CRD types:

```sh
make manifests generate
cp config/crd/bases/cache.wellcake.io_*.yaml charts/valkey-operator/crds/
```

## Common make targets

- `make manifests` — regenerate CRDs, RBAC, and webhook configs from
  kubebuilder markers.
- `make generate` — regenerate `zz_generated.deepcopy.go`.
- `make install` / `make uninstall` — apply / remove the CRDs against the active
  kubectl context.
- `make run` — run the controller out-of-cluster (uses your
  kubeconfig).
- `make docker-build IMG=...` / `make docker-push IMG=...` — build/push
  the image.
- `make deploy IMG=...` / `make undeploy` — apply the `config/default`
  kustomize bundle.
- `make test` — unit + envtest suite (the envtest binaries are fetched by
  `make envtest`).

## Running the controller locally

```sh
make install            # installs the CRDs into the active cluster
make run                # runs the manager on your laptop, watching the cluster
```

In another terminal:

```sh
kubectl apply -f config/samples/cache_v1alpha1_valkeycluster.yaml
kubectl get valkeyclusters -w
```

## Adding a field to ValkeyCluster

1. Edit `api/v1beta1/valkeycluster_types.go` — this is the hub/storage version
   (`+kubebuilder:storageversion`) that the controller reconciles. **Mirror the
   same field in `api/v1alpha1/valkeycluster_types.go`** — this is the served
   spoke that an upstream GitOps layer may write to; otherwise admission prunes
   the unknown field.
2. Add kubebuilder markers (`+kubebuilder:default=...`, `+optional`,
   validation markers,
   `+kubebuilder:validation:XValidation:rule=..."`).
3. `make manifests generate`.
4. Use the field in the builders
   (`internal/controller/resources.go` and neighbors).
5. Sync the chart's CRDs:
   `cp config/crd/bases/* charts/valkey-operator/crds/`.
6. `go build ./...` + `go vet ./...`.
7. Add it to `docs/crd-reference.md`.

## Adding a new topology

1. Extend the `Topology` enum and `+kubebuilder:validation:Enum` in
   `api/v1beta1/valkeycluster_types.go` (hub), then mirror it in `api/v1alpha1/`.
2. Update the validation rules so that conditional fields are required
   iff this topology is selected.
3. Add a new `reconcileXxx` method or file in
   `internal/controller/`.
4. Wire it into the dispatcher in `Reconcile`.
5. Update the Helm chart values.yaml (`charts/valkey-cluster/values.yaml`)
   and the `valkeycluster.yaml` template.
6. Add a section to `docs/topologies.md`.

## Adding a CRD

```sh
kubebuilder create api --group cache --version v1alpha1 --kind <Name> --resource --controller
```

Then:
1. Replace the scaffolded `Spec`/`Status` with the real fields.
2. Replace the scaffolded reconciler.
3. The new controller is wired into `cmd/main.go`
   automatically by kubebuilder.
4. Add RBAC permissions for external resources if needed.
5. Sync the CRD into `charts/valkey-operator/crds/`.
6. Document it in `docs/crd-reference.md`.

## Tests

> The full test map (levels, what is covered, how to run, e2e) —
> [docs/testing.md](testing.md). Below is a short digest for developers.

Pure-function unit tests for the builders live in `internal/controller/`
(see `resources_test.go`). Run them with:

```sh
go test ./internal/controller/ -run 'TestApply|TestTotal|TestStatefulSet|...' -count=1
```

They do not require envtest, do not start an API server — they compile and
run in ~1s.

Reconciler-level tests using the controller-runtime fake client live in
`reconciler_test.go`. Also without envtest:

```sh
go test ./internal/controller/ -run 'TestReconcile' -count=1
```

The full envtest suite (`suite_test.go`) from kubebuilder pulls in etcd +
kube-apiserver:

```sh
make envtest && make test
```

For new logic, prefer table-driven tests in `*_test.go` next to the
file being changed — they compile and run fast.

## Release

1. Bump `version` and `appVersion` in both `charts/*/Chart.yaml`.
2. CI builds and publishes the image to the
   `ghcr.io/melancholictheory/wellcake` registry.
   Locally, by hand:
   `make docker-build docker-push IMG=ghcr.io/melancholictheory/wellcake:<tag>`.
3. Tag the repository, push.
4. Add an entry to `CHANGELOG.md` under the new version; move
   `[Unreleased]` here.
5. (Once a chart repository exists) `helm package` + `helm repo index`
   + push.

## Style notes

- Comments explain **why**, not what. The what is already visible in the code.
- Builders take a `*cachev1alpha1.ValkeyCluster` and return a
  typed k8s object; they are pure functions and never call the
  API server.
- The `ensureFoo(ctx, vc)` methods on the reconciler perform read/create/update
  side effects. Builders are paired with `applyService` / `applyXxx` helpers
  in the reconciler.
- Conditions follow `metav1.Condition` semantics; use the
  `ConditionXxx` constants from the api package; `setCondition` updates
  `LastTransitionTime` only when the status flips (do not bypass this
  directly).
